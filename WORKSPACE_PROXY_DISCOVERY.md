# Domino Workspace Proxy: Discovery & Architectural Implications

*Discovered during development of Domino Databases — June 2026*

---

## The Discovery

While adding IDE workspace tools (Jupyter, JupyterLab, VSCode, RStudio) to the `dd-dse-db`
environment via the Domino environments API, we configured VSCode with:

```python
"proxyConfig": {
    "internalPath": "/",
    "port": 8888,
    "rewrite": False,
    "requireSubdomain": False,
}
```

When a user opened a workspace with the `dd-dse-db` environment and selected VSCode as their
IDE, instead of VSCode appearing in the IDE panel — the entire Domino website loaded. The
workspace panel **was** Domino. Navigating it worked normally. It was indistinguishable from
opening Domino in a new tab.

JupyterHub worked fine. VSCode (and by extension any tool configured this way) turned the
workspace into a pass-through window onto Domino itself.

This same behavior was apparently observed once ~6 months prior and not fully understood at
the time.

---

## The Mechanism, Step by Step

### How Domino workspace proxying works

When a workspace starts, Domino's notebook session infrastructure binds the IDE tool to an
externally accessible URL. There are two routing modes, controlled by `requireSubdomain`:

**Subdomain mode** (`requireSubdomain: true`):
```
https://{runId}.notebookSession.cloud-dogfood.domino.tech/
```
The IDE gets its own subdomain. All paths (`/`, `/static/`, `/api/`) are relative to that
subdomain root. Absolute URLs generated by the IDE resolve correctly because the subdomain
IS the IDE.

**Path mode** (`requireSubdomain: false`):
```
https://cloud-dogfood.domino.tech/u/{owner}/{project}/notebookSession/{runId}/proxy/8888/
```
The IDE is accessible under a path prefix on the main Domino domain. Domino strips that
prefix and forwards the request to port 8888 inside the container.

### Why Jupyter works in path mode

Jupyter's `internalPath` is set to the full handlebars-templated proxy path:

```
/{{ownerUsername}}/{{projectName}}/{{sessionPathComponent}}/{{runId}}/
```

Domino's workspace infrastructure injects this path into JupyterHub's startup via
`--NotebookApp.base_url=<path>`. JupyterHub then generates every link, asset URL, and API
call relative to that base. A link that Jupyter renders as `/u/nick/proj/.../tree/` actually
IS the right external URL, so it works.

### Why VSCode breaks (and Domino appears)

VSCode (code-server) doesn't receive any base-path configuration in our setup — its
`internalPath` is `"/"`. The start script at `/opt/domino/workspaces/vscode/start` would
need the full proxy path injected to configure code-server's `--base` flag, but with
`internalPath: "/"` that information isn't available.

code-server starts on port 8888, blissfully unaware it's behind a `/u/nick/.../proxy/8888/`
prefix. When its HTML renders, it generates absolute asset references:

```html
<script src="/static/out/vs/workbench/workbench.web.main.js"></script>
<link rel="stylesheet" href="/static/...">
```

The browser fetches those absolute paths from `https://cloud-dogfood.domino.tech/static/...`.
That is NOT VSCode's static directory. That is Domino's own static file server.

The page that renders is therefore assembled partly from VSCode's HTML scaffold and entirely
from Domino's CSS, JS, and assets. The result: the browser renders the Domino UI. Completely.
Navigating it works because every link still points at Domino and the user's session cookie is
present. It is functionally Domino running inside the workspace panel.

### The key properties that make this work

1. **Same-origin**: The workspace IDE iframe is served from `cloud-dogfood.domino.tech`, the
   same origin as Domino itself. No CORS. No cross-origin restrictions.

2. **Authenticated session**: The user's Domino session cookie is present in every request
   that flows through the IDE iframe. The browser doesn't distinguish "workspace requests" from
   "Domino requests" — they're all the same origin.

3. **Auth proxy**: Inside the container, `localhost:8899` is the Domino auth proxy. Any HTTP
   request the workspace app makes to this proxy is automatically signed with the user's
   identity and forwarded to Domino's API. No token management required.

4. **Port 8888 is the IDE slot**: Whatever is listening on port 8888 in the container when a
   workspace is open IS the IDE, from Domino's perspective. The workspace infrastructure
   doesn't care what it is.

5. **The chain**:
   ```
   browser ──► Domino proxy ──► container:8888 ──► your app ──► localhost:8899 ──► Domino API
   ```
   Your app sits at the exact midpoint between the user's browser and Domino's backend, with
   full auth on both sides.

## What This Unlocks: Architectural Patterns

### Pattern 1: Domino Databases as a Workspace IDE

**The immediate application.** Add a `"domino-databases"` entry to `DEFAULT_WORKSPACE_TOOLS`
with `internalPath: "/"` pointing at the DB wizard (`app.py`) running on port 8888.

A user opens a workspace, selects "Domino Databases" as their IDE, and gets the full DB
management UI embedded in the workspace panel — not in a separate App, not in a new tab, but
as a native-feeling pane. Since they're already in a workspace with the auth proxy available,
all API calls just work. No tunnel, no extra auth, no separate App product.

This means database management could be a first-class workspace experience. Create a DB,
monitor it, snapshot it, all from inside your development environment.

The implementation delta is small: the wizard already runs as a Flask app; it just needs a
workspace tool entry and a `gunicorn` launcher in `app.sh` that binds to port 8888 when
`DD_ROLE=wizard` is detected in a workspace context.

### Pattern 2: Transparent Domino UI Enhancement Layer

**The middleware play.** Run a Flask (or any HTTP) app on port 8888 that:
- Proxies the vast majority of requests upstream to Domino's real backend
- Intercepts specific routes to inject custom HTML, JS, or CSS into responses
- Operates with full same-origin trust — the browser never knows it isn't talking to Domino

Because the app is same-origin, JavaScript it injects can call `fetch("/api/v4/...")` directly
against Domino's real API. The user's session cookie is automatically sent. No OAuth flow, no
API key management, no CORS preflight.

What you could build with this:
- **Custom search**: intercept Domino's search page, enhance it with vector similarity or
  project-aware filtering
- **Keyboard shortcuts**: inject a JS layer that adds VSCode-style command palette behavior
  to any Domino page
- **Cross-project navigation**: inject a sidebar that persists across Domino page navigations
  (normally impossible because Domino's SPA resets state on route changes)
- **Audit layer**: log every API call the user makes through the workspace for compliance or
  debugging without touching Domino's backend

The proxy app would look something like:
```python
@app.route("/<path:path>", methods=["GET","POST","PUT","DELETE","PATCH"])
def proxy(path):
    # Forward to real Domino
    resp = requests.request(method, f"http://localhost:8899/{path}", ...)
    if "text/html" in resp.headers.get("Content-Type",""):
        # Inject your custom JS/CSS
        body = resp.text.replace("</head>", "<script src='/custom.js'></script></head>")
        return body
    return resp.content, resp.status_code, dict(resp.headers)
```

This is a legitimate Domino extension system that Domino itself doesn't ship.

### Pattern 3: Domino DevTools / Admin Panel

**The same-origin API explorer.** A workspace that runs a custom admin/debugging UI on port
8888. It has:
- A live Domino API explorer (call any `/v4/...` endpoint, see formatted JSON responses)
- A project-scoped filesystem browser with git status awareness
- Live environment build log viewer
- App/job monitoring dashboard with auto-refresh

Because it runs in the workspace with the auth proxy, every API call is pre-authenticated.
The user doesn't configure tokens or base URLs — the tool just works against their current
Domino context.

This is particularly powerful for platform engineers and admins who need visibility into
Domino internals that the standard UI doesn't expose cleanly.

### Pattern 4: "Sidecar" Workspace App

**Parallel to your real IDE.** Rather than replacing your IDE, a sidecar runs as a second
process. You open two browser tabs: one at the normal workspace path (JupyterLab or VSCode
via subdomain, working correctly), and one at the `proxy/8888/` path (the sidecar).

The sidecar could be:
- A live documentation server that knows which notebook you have open
- A dataset browser synchronized with your workspace filesystem
- A model/experiment tracker that auto-logs cells as you run them

The sidecar shares the container's filesystem with the main IDE process, so it can watch
files, read outputs, and write back without any network hop.

## The Underlying Insight

The standard mental model of a Domino workspace is:

```
User → Domino → [container running Jupyter/VSCode]
```

The actual model (with this discovery) is:

```
User → Domino → [container running ANYTHING on :8888] → Domino (via auth proxy)
```

The workspace is not a locked-down sandbox running a specific IDE. It's an authenticated,
same-origin, bidirectional proxy slot. Domino routes browser traffic TO it, and it can make
authenticated API traffic back THROUGH it. The slot happens to be occupied by Jupyter or
VSCode by default. It doesn't have to be.

The `requireSubdomain: false` + `internalPath: "/"` combination — which accidentally made
VSCode show the entire Domino website — is actually the configuration that enables all of
these patterns. With subdomain routing, you get isolation. With path routing and `/` as the
internal path, you get same-origin identity. The "bug" is the feature.

## Prior Art and Context

This was stumbled upon once ~6 months before this writeup, also accidentally. The behavior
is surprising enough that it wasn't recognized as an architectural primitive at the time —
it looked like a bug (IDE not loading correctly) rather than a capability (full same-origin
proxy access to Domino).

The combination of same-origin + pre-authenticated + bidirectional (browser-facing AND
API-calling) is rare. Most "embedded app" patterns involve at minimum a CORS header dance or
a separate auth flow. Here, both sides are automatically trusted because of where the
container sits in the network topology.

Domino likely didn't design this intentionally. It's an emergent property of how workspace
proxying is implemented that becomes a significant extension mechanism once understood.
